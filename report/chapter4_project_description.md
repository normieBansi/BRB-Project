Chapter 4: Project Description
4.1 System Overview
The Beyond Rule Based Next Generation Firewall (BRB-NGFW) is an open source network security architecture designed to augment conventional stateful firewalls with machine learning driven anomaly detection and automated enforcement. Built upon OPNsense, a production grade firewall distribution, the system utilizes a sidecar architecture. This approach allows the intelligence layer to operate as an external process, avoiding modifications to the core firewall codebase. The resulting platform enforces traditional access control through the OPNsense packet filter, conducts signature based detection via Suricata, identifies statistical anomalies using a trained Isolation Forest model, and automatically executes blocking decisions through the OPNsense REST API. Crucially, this is achieved without requiring proprietary components or commercial licensing. The system was developed and validated in a virtualized three segment laboratory environment. This setup included a WAN facing router, a LAN segment hosting a vulnerable Ubuntu target, and an OPT1 attacker segment housing a Kali Linux machine utilized for executing ten distinct attack scenarios.
4.2 System Objectives
The primary objective of the BRB-NGFW project is to demonstrate that adaptive, behavior based threat detection can be seamlessly integrated into an open source firewall platform using standard tools. Specifically, the system aims to detect network anomalies invisible to static rule based enforcement and provide coverage for attack patterns lacking Suricata signatures. Furthermore, the architecture is designed to translate detection outputs into automated enforcement actions, drastically reducing mitigation latency. Maintaining full operator visibility and auditability over all automated decisions remains a critical priority. These objectives collectively define an adaptive, automated, and transparent security posture, qualities traditionally restricted to expensive commercial platforms.
4.3 Sidecar Architecture
The defining structural choice in the BRB-NGFW project is the implementation of the sidecar pattern. Rather than embedding custom code within the firewall, the machine learning engine, control API, and observability stack are deployed as separate processes on an adjacent Ubuntu control node. OPNsense continues to perform its core stateful filtering and Suricata inspection unmodified. The sidecar components consume the firewall's telemetry outputs and issue REST API calls to orchestrate enforcement. This architecture was selected for three fundamental reasons. First, it ensures upgrade compatibility, allowing OPNsense and Suricata to be updated independently of the machine learning pipeline. Second, it eliminates per packet processing overhead on the firewall, as the inference engine operates on separate compute resources. Third, it maintains a strict boundary between the detection and enforcement planes. The firewall rule engine remains the authoritative source of truth, with the machine learning system functioning purely as an orchestration layer.
4.4 Module Breakdown
4.4.1 Log Collection
Efficient telemetry ingestion forms the foundation of the analytics pipeline. OPNsense is configured to forward firewall and IDS log events to the control node using syslog-ng over UDP port 514. Promtail, a robust log shipping agent, reads this incoming stream and forwards all log entries to Loki for backend aggregation. This pipeline ensures logs are persistently stored for historical analysis while remaining available in near real time for inference. Simultaneously, Suricata events are logged locally in JSON format, providing rich contextual data for the parsing module.
4.4.2 Parsing and Feature Extraction
The parsing module, implemented in Python, normalizes heterogeneous data from OPNsense filterlogs and Suricata alerts into a unified flat schema. Crucially, feature engineering focuses exclusively on metadata, extracting elements such as IP addresses, port numbers, protocol types, and firewall actions. Payload content is intentionally ignored. This metadata centric approach ensures the detection model remains structurally resilient to the increasing ubiquity of TLS 1.3 encryption, which fundamentally degrades the effectiveness of traditional deep packet inspection.
4.4.3 ML Training and Inference
The machine learning pipeline utilizes scikit-learn to construct and evaluate the anomaly detection model. The training script applies a preprocessing pipeline consisting of a StandardScaler for numerical features and a OneHotEncoder for categorical features. An Isolation Forest model, configured with 200 estimators, is then fitted to the baseline traffic data. The Isolation Forest was selected because its unsupervised nature eliminates the need for labeled attack data, making it highly adaptable to diverse network environments. During inference, the serialized model evaluates incoming log records, producing continuous anomaly scores. Records breaching the defined anomaly threshold are immediately forwarded to the FastAPI control service.
4.4.4 API Control Layer
The control service, built using the FastAPI framework, functions as the crucial orchestration layer between the detection engine and the firewall. Upon receiving an anomaly notification, the service issues an authenticated POST request to the OPNsense REST API, dynamically adding the adversarial source IP to the AUTO_BAN_IPS alias. A subsequent reconfiguration request is triggered to apply the block rule immediately. To ensure operational transparency, the service maintains a comprehensive local audit log recording every automated ban decision, including the triggering anomaly score and the firewall's API response.
4.4.5 Observability Dashboard
The observability stack, deployed via Docker Compose, provides real time visualization of the system's operational state. Grafana connects to the Loki data source to render custom dashboards displaying firewall event volumes, Suricata alert frequencies, and active anomaly detections. A dedicated panel continuously polls the OPNsense API to display the current contents of the block list. This unified interface demystifies the automated enforcement loop, allowing operators to rapidly verify detection accuracy and monitor the system's response behavior.
4.5 End-to-End Workflow
The operational workflow proceeds through a rapid, automated sequence. Network traffic initially traverses the OPNsense firewall, generating filterlog entries and potential Suricata alerts. These logs are forwarded via syslog-ng to the control node, where the parsing module extracts the relevant metadata features. The inference pipeline immediately evaluates these features against the trained Isolation Forest model. If a flow is classified as an anomaly, the script asynchronously triggers the FastAPI control service. The service executes the necessary REST API calls to update the firewall's block alias, successfully neutralizing the threat. This entire pipeline from packet arrival to automated enforcement operates within a latency budget of a few seconds. This achievement represents a profound operational improvement over the manual response models prevalent in standard open source deployments.
