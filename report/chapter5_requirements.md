Chapter 5: Requirements

5.1 Functional Requirements

Functional requirements specify the behaviours that the BRB-NGFW system must exhibit in order to fulfil its stated objectives. Each requirement is described in terms of the specific input, processing, and output expected from the corresponding system component, grounded in the actual implementation.

FR-01: Log Ingestion
The system must continuously receive and store firewall and intrusion detection log events generated by OPNsense and Suricata. OPNsense filterlog entries must be forwarded to the control node via syslog-ng over UDP port 514. Suricata alert events must be written to /var/log/suricata/eve.json in JSON format. Promtail must read both log streams and ship them to the Loki log aggregation backend without data loss. The log ingestion pipeline must be capable of sustaining continuous operation without manual restarts and must preserve the original timestamp of each event.

FR-02: Feature Extraction
The system must parse raw log records from both OPNsense filterlog and Suricata eve.json formats and normalise them into a unified flat schema. The parser (ml/parse_logs.py) must extract the fields: event_source, src_ip, dst_ip, dst_port, proto, action, direction, rule_label, suricata_signature, and suricata_category from each record. Fields absent in a given record must be filled with a designated sentinel value to maintain consistent column cardinality. The parsed output must be written to ml/data/parsed_logs.csv and must be readable by both the training and inference pipelines without additional transformation.

FR-03: Model Training
The system must be capable of training an Isolation Forest anomaly detection model on a dataset of parsed log records representing baseline network behaviour. The training pipeline (ml/train.py) must apply a preprocessing stage consisting of StandardScaler for numerical features and OneHotEncoder with unknown category handling for categorical features. The trained model and preprocessing pipeline must be serialised together as a single artefact to ml/models/isolation_forest.pkl using joblib. The training pipeline must accept a configurable contamination parameter and must produce a deterministic model when a fixed random seed is specified.

FR-04: Real-Time Inference
The system must apply the trained Isolation Forest model to incoming parsed log records and produce an anomaly prediction and score for each record. The inference pipeline (ml/infer.py) must load the serialised model, apply the same preprocessing transformation used during training, and classify each record as anomalous or normal. Records classified as anomalous must be written to ml/data/anomalies.csv with their associated anomaly scores and source IP addresses. The inference pipeline must be capable of processing new records without requiring model retraining.

FR-05: Automated Blocking
The system must automatically translate anomaly detection outputs into firewall enforcement actions without requiring human intervention. The FastAPI control service (control-api/app.py) must expose a /ban endpoint that accepts a JSON payload containing a source IP address and anomaly score. On receiving a valid request, the control service must add the specified IP address to the OPNsense AUTO_BAN_IPS alias by calling the OPNsense REST API alias management endpoint. The control service must subsequently trigger an alias reconfiguration call to ensure the block is applied to the active packet filter rule set within seconds of the detection event. All ban decisions must be recorded in an audit log with timestamp, source IP, anomaly score, and API response status.

FR-06: Dashboard Visualisation
The system must provide a real-time monitoring interface that displays firewall event volume, Suricata alert frequency, current anomaly detections, and the active block list. The Grafana dashboard must query the Loki data source for log-based panels and the OPNsense REST API for the current contents of the AUTO_BAN_IPS alias. The dashboard must refresh automatically and must present all panels without requiring manual user interaction beyond navigating to the Grafana URL.

5.2 Non-Functional Requirements

Non-functional requirements specify the quality attributes that govern how the BRB-NGFW system must perform its functions. They define the operational constraints within which the functional requirements must be satisfied.

NFR-01: Scalability
The system must be capable of processing log volumes commensurate with the traffic levels generated in the virtualised laboratory environment without degrading inference latency or log pipeline throughput. The sidecar architecture must ensure that increases in firewall rule complexity or Suricata ruleset size do not affect the performance of the ML inference or control service components, as these operate on separate compute resources. The parsing and inference pipeline must be designed to accommodate additional log sources or feature columns with changes confined to the parsing module, without requiring retraining of the Isolation Forest model architecture.

NFR-02: Performance
The end-to-end latency from a log event being generated by OPNsense or Suricata to the corresponding block decision being applied to the active firewall rule set must not exceed thirty seconds under normal operating conditions. The Isolation Forest inference step must complete within two seconds for a batch of up to one thousand log records on the control node hardware. The OPNsense REST API call sequence (alias addition followed by reconfiguration) must complete within five seconds of the control service receiving a valid ban request. The Grafana dashboard must render all panels within ten seconds of a page load.

NFR-03: Reliability
The log ingestion pipeline must not lose log events under sustained normal traffic load. Promtail must maintain a position file to enable recovery from temporary Loki unavailability without data loss. The FastAPI control service must return a meaningful error response if the OPNsense API is unreachable, and must record the failed attempt in the audit log. The system must continue to perform signature-based detection through Suricata and rule-based enforcement through OPNsense even if the ML pipeline or control service is unavailable, ensuring that the sidecar failure does not degrade the baseline security posture of the firewall.

NFR-04: Maintainability
All system components must be independently deployable and independently restartable without affecting the operation of other components. The ML training and inference pipelines must be re-executable at any time to update the model with new baseline data, with the updated model taking effect on the next inference pipeline invocation. Configuration parameters, including OPNsense API credentials, the contamination threshold, the Loki endpoint URL, and the syslog listener port, must be stored in environment variable files and must not be hardcoded in source files. The codebase must be organised in clearly named directories (ml/, control-api/, docker/) with no circular dependencies between modules.

NFR-05: Security
OPNsense API credentials used by the control service must be stored in a .env file that is excluded from version control. The control service must authenticate to the OPNsense API using a dedicated, least-privilege API key that has access only to the alias management endpoints required for automated blocking. Communication between the control service and the OPNsense API must use HTTPS. The AUTO_BAN_IPS alias and its associated firewall rule must be the only mechanism through which the ML pipeline influences the packet filter rule set; no other OPNsense configuration must be modifiable by the control service. The Grafana interface must be accessible only on the internal LAN segment and must require authentication before displaying dashboard panels.
