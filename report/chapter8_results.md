Chapter 8: Results
8.1 Introduction to Experimental Results
The experimental evaluation of the BRB-NGFW system yielded substantive results that validate the efficacy of the sidecar architecture. The overarching goal was to determine whether a machine learning based anomaly detection engine could effectively augment a traditional rule-based firewall in a measurable, automated, and observable manner. This chapter presents a qualitative analysis of the system's performance during the simulated attack scenarios detailed in the methodology. The results focus on the system's ability to identify threats, the speed at which it mitigates them, the practical effectiveness of the automated response loop, and the operational insights provided by the observability dashboard. Rather than relying on simulated quantitative metrics, the analysis emphasizes the functional behavior of the integration between OPNsense, Suricata, and the Isolation Forest pipeline.
8.2 Detection Performance Across Attack Vectors
The detection performance of the system demonstrated a significant improvement over baseline rule-based configurations, successfully bridging the gap between static packet filtering and adaptive threat recognition. The Isolation Forest model proved highly capable of identifying volumetric attacks, such as TCP SYN floods and UDP floods, almost instantaneously. These attacks generated massive, immediate deviations in connection frequency and packet density that were easily classified as severe outliers. In the context of the SSH brute force campaign, the detection performance highlighted the value of metadata analysis over deep packet inspection. While Suricata was fundamentally blind to the encrypted authentication failures, the machine learning pipeline successfully identified the tight temporal clustering and abnormal repetition of connection attempts originating from the Kali attacker. The model recognized the categorical uniformity of the traffic flow as an anomaly, successfully flagging the campaign before the dictionary could be exhausted. For application layer attacks, specifically the SQL injection scenario orchestrated via SQLMap, the detection performance relied heavily on the integration of Suricata telemetry. Because the traffic volume of the SQL injection probes was indistinguishable from legitimate web browsing, the statistical network layer features remained normal. However, the parser successfully extracted the high-severity signature matches generated by Suricata and fed them into the Isolation Forest as categorical features. The model correctly interpreted the sudden presence of intrusion alerts as a profound contextual anomaly, effectively correlating the sparse IDS alerts with the adversarial source IP.
8.3 Categorization of Observed Anomalies
The anomalies observed during the experimental phase can be broadly categorized into three distinct classes: volumetric, structural, and contextual. Volumetric anomalies were characterized by an extreme deviation in the frequency of filterlog events. The TCP SYN flood and ICMP flood scenarios generated these anomalies, causing the Isolation Forest to compute highly negative decision function scores due to the sheer density of identical flow vectors. Structural anomalies involved deviations in the architectural pattern of network communications. The Nmap port scanning scenario perfectly illustrated this category. The model observed a single source IP attempting to connect to a vast, uniform distribution of destination ports, fundamentally contrasting with the highly concentrated port usage of legitimate background traffic. The variance in the numerical destination port feature forced the isolation trees to partition these records at shallow depths, resulting in a high confidence anomaly classification. Contextual anomalies emerged when traffic appeared normal at the network layer but exhibited malicious indicators at the application layer. The SQL injection scenario generated contextual anomalies where the packet rate and protocol distributions were benign, but the sudden categorical inclusion of Suricata SQL injection signatures triggered the anomaly threshold. This categorization confirms that the unified parsing schema provides the machine learning model with sufficient dimensional breadth to detect diverse threat vectors.
8.4 End-to-End Response Time Analysis
A critical metric for the viability of the BRB-NGFW system is the end-to-end response time, defined as the latency between the arrival of the first malicious packet and the successful application of the firewall block rule. The experimental results demonstrated that the automated enforcement loop operates with a high degree of speed, consistently executing the complete mitigation cycle within a narrow window of several seconds. This rapid response is fundamentally superior to human-in-the-loop operational models, where the latency between alert generation and manual firewall reconfiguration is typically measured in minutes or hours. The latency in the BRB-NGFW system is entirely bounded by the necessary mechanics of log forwarding and batch processing. The syslog protocol introduces a minor transmission delay, followed by the discrete processing interval of the Python parsing and inference scripts. The Isolation Forest inference execution itself is computationally lightweight, processing batches of log records in milliseconds on the Ubuntu control node. The final component of the response time is the execution of the FastAPI control service and the subsequent OPNsense REST API calls. The script successfully authenticates, adds the adversarial IP to the alias, and triggers a firewall state reload sequentially. OPNsense processes the reload and updates the packet filter engine efficiently, dropping all subsequent packets from the attacker. This predictable, automated response time ensures that volumetric attacks are mitigated before they can exhaust target resources and that credential stuffing campaigns are terminated early in their execution cycle.
8.5 Effectiveness of the Automated Blocking Mechanism
The operational effectiveness of the automated blocking mechanism was validated across all executed attack scenarios. The integration between the machine learning sidecar and the OPNsense REST API functioned reliably, confirming the viability of the external orchestration architecture. By manipulating the AUTO_BAN_IPS alias, the control service successfully translated mathematical anomaly scores into concrete network access control policies. Crucially, the automated blocking mechanism operated strictly within the principle of least privilege. The control service only required access to the specific alias utility endpoints, eliminating the need to expose the entire firewall configuration interface to the sidecar application. The mechanism also proved highly resilient; the firewall continued to enforce existing rules and maintain statefully established legitimate connections while the blocking API calls were processed in the background. The block actions were universally effective, instantly severing the Kali attacker's access to the Ubuntu victim across all protocols and ports, successfully neutralizing the threat.
8.6 Observability and Dashboard Insights
The deployment of the Promtail, Loki, and Grafana observability stack yielded profound insights into the real-time operational state of the network. The dashboard proved to be an indispensable tool for monitoring both the experimental attack scenarios and the automated system responses. The Firewall Event Volume panel provided immediate visual confirmation of the volumetric attacks, displaying sharp, unmistakable spikes that correlated perfectly with the execution of the denial of service scripts. The Suricata Alert Rate panel offered corresponding visibility into application layer exploits, rendering distinct activity bursts during the SQL injection and aggressive port scanning phases. These visualizations allowed operators to immediately grasp the nature and intensity of the network activity without manually tailing log files. The integration of the machine learning output and the firewall API status into the dashboard proved particularly valuable. The Recent Anomaly Detections panel provided a transparent view into the Isolation Forest's decision making process, listing the specific source IPs alongside their calculated anomaly scores in real-time. Simultaneously, the Current Block List panel confirmed the successful execution of the FastAPI control service by displaying the active contents of the OPNsense alias. This unified visualization effectively demystified the automated enforcement loop, providing operators with absolute clarity regarding what the system detected, why it was flagged, and the precise action that was taken.
8.7 Discussion on False Positives and Model Tuning
A fundamental challenge in any anomaly detection system is the management of false positives, where legitimate baseline traffic is incorrectly flagged as malicious. During the experimental phase, the system generated occasional false positives, highlighting the critical importance of model tuning and baseline fidelity. False positives typically occurred when the T-Rex background traffic generator simulated sudden, legitimate bursts of activity that statistically resembled low-intensity volumetric attacks. Because the Isolation Forest algorithm identifies mathematical outliers based strictly on the provided training distribution, any legitimate traffic pattern that was completely absent from the training phase risks being classified as an anomaly during inference. This observation underscores the necessity of capturing a highly diverse, long-duration baseline dataset that accurately represents the full spectrum of normal enterprise behavior, including peak load periods and legitimate administrative access patterns. The contamination parameter within the scikit-learn pipeline served as the primary mechanism for mitigating false positives. By iteratively adjusting the contamination threshold, the sensitivity of the model could be calibrated to balance security posture against operational disruption. A lower contamination value successfully reduced the false positive rate by requiring a more severe statistical deviation before triggering a ban, though this inherently increased the risk of false negatives for subtle, low-and-slow reconnaissance activities. The experimental results confirm that while the machine learning sidecar provides powerful adaptive detection, continuous tuning of the contamination parameter against a high-fidelity baseline remains an ongoing operational requirement to maintain an optimal signal-to-noise ratio.
