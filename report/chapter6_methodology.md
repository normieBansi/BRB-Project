Chapter 6: Methodology
6.1 Overview of the Methodological Approach
This chapter presents the complete technical methodology underlying the BRB-NGFW system. Methodology in this context refers to the systematic sequence of design decisions, configuration steps, implementation choices, and operational procedures that together constitute the working system. The presentation is structured to follow the data flow of the system: from the network traffic that enters the firewall, through the logging and parsing pipeline, through model training and inference, through automated enforcement, and finally to the observability layer that provides operator visibility over the entire process.
Each section documents not only what was done but why the specific approach was taken in preference to alternatives, grounding the methodological choices in the technical constraints of the deployment environment and the research objectives of the project. Where specific configuration values or code constructs are referenced, they correspond to the actual implementation in the project repository.
6.2 System Architecture
The BRB-NGFW system is organised across four functional planes: the Data Plane, the Detection Plane, the Analytics and Telemetry Plane, and the Adversarial Simulation Plane. This four-plane model is not a formal architectural specification but a conceptual framework for understanding the responsibilities and boundaries of the system components.
The Data Plane encompasses the OPNsense firewall and its packet filter (pf) engine. This plane is responsible for forwarding or blocking all network traffic in the lab environment based on the configured rule set and the current contents of alias tables. The Data Plane is the enforcement point: every block decision ultimately manifests as a change to the active state of this plane. The Data Plane is intentionally kept as a standard, unmodified OPNsense deployment. No custom code is introduced into OPNsense itself. All interactions with the Data Plane from other components occur exclusively through the OPNsense REST API.
The Detection Plane encompasses Suricata, the intrusion detection and prevention engine integrated with OPNsense. Suricata performs deep packet inspection against the Emerging Threats Open signature ruleset and generates structured alert events in JSON format for any traffic that matches a loaded signature. In the BRB-NGFW deployment, Suricata operates in IDS mode (alert-only, not drop-mode) because enforcement decisions are delegated to the Data Plane through the ML control pathway rather than handled by Suricata directly. This design ensures that all block decisions are audited and reversible through the alias management interface.
The Analytics and Telemetry Plane encompasses the ML pipeline (parse_logs.py, train.py, infer.py), the FastAPI control service (control-api/app.py), Promtail, Loki, and Grafana. This plane is responsible for consuming telemetry from the Data and Detection Planes, deriving statistical characterisations of network behaviour, producing anomaly predictions, and issuing enforcement instructions back to the Data Plane. This plane runs entirely on the Ubuntu control node at 192.168.50.20, physically separate from the OPNsense appliance.
The Adversarial Simulation Plane encompasses the Kali Linux attacker machine at 192.168.60.10 and the scripts in the kali-scenarios directory. This plane exists solely for the purpose of generating known-malicious traffic to validate the detection and enforcement capabilities of the other planes during experimental evaluation.
The physical network topology of the lab consists of three segments: a WAN interface connecting OPNsense to the external network, a LAN interface (192.168.50.0/24) connecting OPNsense to the Ubuntu victim machine (192.168.50.10) and the Ubuntu control node (192.168.50.20), and an OPT1 interface (192.168.60.0/24) connecting OPNsense to the Kali attacker machine (192.168.60.10). This topology enforces that all traffic between the attacker and victim segments must traverse the OPNsense firewall, ensuring that all attack traffic is visible to both the pf engine and Suricata.
6.3 OPNsense Configuration and Logging
OPNsense serves as both the enforcement point and the primary telemetry source for the BRB-NGFW system. The configuration of OPNsense for this project involved four distinct areas: interface and network topology setup, firewall rule configuration, Suricata IDS integration, and syslog forwarding.
6.3.1 Interface and Network Configuration
Three network interfaces are configured in OPNsense: the WAN interface with DHCP addressing for external connectivity, the LAN interface with the static subnet 192.168.50.0/24, and the OPT1 interface with the static subnet 192.168.60.0/24. OPNsense is configured to perform NAT on the WAN interface for outbound traffic from both LAN and OPT1 segments. The firewall rules on the LAN and OPT1 interfaces permit inter-segment routing between the two internal networks, which is required to allow the Kali attacker to reach the Ubuntu victim across the OPNsense routing boundary. Default OPNsense behaviour blocks all inbound traffic not associated with an established connection; this default-deny posture is preserved for WAN-facing traffic.
6.3.2 Firewall Rule Configuration
The firewall rule set is configured to reflect a realistic but permissive internal policy, representative of a small organisation that allows broad inter-segment connectivity. Rules on the OPT1 interface permit outbound traffic on TCP ports 22 (SSH), 80 (HTTP), 443 (HTTPS), and all ICMP and UDP traffic, allowing the full range of attack scenarios to reach the target host. This permissive policy is intentional: it represents the realistic condition under which signature-based and anomaly-based detection must operate, where the access policy cannot simply enumerate all legitimate traffic because the definition of legitimate traffic is itself broad.
A critical configuration element is the AUTO_BAN_IPS alias. This is a host-type alias defined in OPNsense under Firewall > Aliases. It is initially empty. A firewall rule is configured on the OPT1 interface with the highest priority (processed first), which blocks all traffic from any source IP listed in the AUTO_BAN_IPS alias. When the ML control service adds an IP address to this alias through the REST API and triggers a reconfiguration, OPNsense immediately begins dropping all traffic from that source, regardless of the other rules on the interface.
6.3.3 Suricata Integration
Suricata is installed in OPNsense through the intrusion detection plugin available in the OPNsense package manager. It is configured to inspect traffic on the OPT1 interface, which is the interface through which all attacker traffic enters the system. The Emerging Threats Open ruleset is downloaded and applied; this ruleset provides coverage for several thousand known attack patterns including web scanning, SQL injection, SSH brute force, and various DDoS signatures. Suricata is configured to write all events to /var/log/suricata/eve.json in extended JSON format, which includes not only alert events but also flow, DNS, HTTP, and TLS metadata events. The ML parsing pipeline reads this file to extract both signature-matched alert events and contextual flow metadata.
6.3.4 Syslog Forwarding
OPNsense is configured to forward its filterlog output to the control node via syslog-ng. In the OPNsense System > Logging > Remote settings panel, the remote syslog target is set to the control node IP address (192.168.50.20) on UDP port 514. The filterlog facility is selected for forwarding, which causes OPNsense to export one syslog line for each packet that is evaluated against the firewall rule set and generates a log entry. Each filterlog line is comma-separated and contains the following positional fields: rule number, sub-rule number, anchor, tracker, interface, reason, action (pass or block), direction (in or out), IP version, and protocol-specific fields including source IP, destination IP, and destination port.
The syslog stream is received on the control node by a syslog-ng listener process configured to write incoming messages to /var/log/opnsense/filterlog.log. This file is read by both the Promtail log shipper (for Loki ingestion) and the parse_logs.py parsing module (for ML feature extraction).
6.4 Log Parsing Process
The parsing module, implemented in ml/parse_logs.py, is responsible for reading raw log data from two heterogeneous sources, OPNsense filterlog and Suricata eve.json, and producing a single unified DataFrame with a consistent schema suitable for ML processing.
6.4.1 OPNsense Filterlog Parsing
The parse_filterlog_line function reads each line from the filterlog file and applies a positional splitting strategy. The filterlog format is structured but variable-length: the number of fields depends on the IP version and protocol of the packet, with IPv4 TCP packets having a different field count from IPv4 UDP packets or IPv6 packets. The parser handles this variability by first extracting the invariant fields at known positions (rule label at index 0, action at index 6, direction at index 7, IP version at index 8) and then branching on IP version and protocol to extract the variable positional fields (source IP, destination IP, destination port, and protocol type). Parsed records are assembled into a dictionary with the keys: event_source (set to "filterlog"), src_ip, dst_ip, dst_port, proto, action, direction, rule_label, suricata_signature (set to "none"), and suricata_category (set to "none").
6.4.2 Suricata eve.json Parsing
The parse_suricata_eve function reads eve.json line by line, parsing each line as a JSON object using Python's standard json library. The parser inspects the event_type field of each record to determine its type. For records with event_type equal to "alert", the parser extracts src_ip, dst_ip, dest_port, proto, and action from the top-level fields, and extracts suricata_signature and suricata_category from the nested alert.signature and alert.category fields respectively. The direction field is inferred from the flow_id and packet direction metadata. For non-alert event types (flow, dns, http, tls), the parser extracts flow-level metadata without signature information and sets suricata_signature and suricata_category to "none". This inclusive approach to non-alert events enriches the training dataset with contextual flow behaviour that would be absent from a purely alert-focused parsing strategy.
6.4.3 Schema Unification and Output
Records from both sources are appended to a shared Python list as dictionaries conforming to the ten-field schema. After all input files are processed, the list is converted to a pandas DataFrame. Any fields that are missing from a specific record type are filled with the string "none" using the pandas fillna method. This sentinel fill strategy was chosen over dropping records with missing fields because dropping would disproportionately affect filterlog records (which have no Suricata signature fields) and would skew the training dataset toward Suricata-heavy representations of traffic. The completed DataFrame is written to ml/data/parsed_logs.csv using pandas to_csv with the index excluded.
6.5 Feature Engineering
Feature engineering in this project is deliberately constrained to metadata-level features derived from the log schema. This constraint is a principled design choice rather than a limitation. Payload-level features, such as byte sequences extracted from packet content, are unavailable for encrypted TLS traffic. As TLS 1.3 adoption has expanded, an increasing proportion of network traffic is opaque to DPI-based feature extraction. By operating exclusively on metadata features, the BRB-NGFW system maintains detection capability across encrypted and unencrypted traffic alike.
The ten features used in the ML pipeline are categorised into two types. Numerical features comprise dst_port, which encodes the destination port as an integer. All remaining features (event_source, src_ip, dst_ip, proto, action, direction, rule_label, suricata_signature, suricata_category) are categorical features represented as strings.
The preprocessing pipeline applies different transformations to each feature type. Numerical features are processed by scikit-learn's StandardScaler, which centres the values to zero mean and scales them to unit variance. This normalisation is important for distance-based computations within the Isolation Forest's tree-building process, ensuring that features with numerically large values (such as high-numbered ephemeral ports) do not disproportionately dominate the partitioning decisions. Categorical features are processed by scikit-learn's OneHotEncoder with handle_unknown set to "ignore". One-hot encoding converts each categorical string value into a binary indicator vector, allowing the Isolation Forest algorithm to treat each unique category value as a distinct dimension in the feature space. The handle_unknown parameter is set to "ignore" rather than raising an error so that categorical values observed during inference but absent during training (for example, a new destination IP not seen in the training data) are handled gracefully by encoding them as all-zero vectors rather than causing a pipeline failure.
The preprocessing transformations for numerical and categorical features are combined into a single scikit-learn ColumnTransformer that applies each transformer to the appropriate column subset. This ColumnTransformer is then combined with the Isolation Forest estimator into a unified scikit-learn Pipeline object, ensuring that the identical preprocessing steps applied during training are automatically applied during inference without any risk of transformation inconsistency.
6.6 Isolation Forest Training Pipeline
The training pipeline is implemented in ml/train.py and proceeds in six steps.
The first step is data loading. The parsed_logs.csv file is read into a pandas DataFrame using pd.read_csv. Records with missing values in the feature columns are dropped, though the sentinel fill strategy in the parsing module ensures this is rare in practice.
The second step is feature column selection. The DataFrame is subset to the ten feature columns defined in the schema. The target variable column (if present) is excluded, as the Isolation Forest is an unsupervised estimator that does not use class labels.
The third step is preprocessing pipeline construction. A ColumnTransformer is instantiated with two transformers: a StandardScaler applied to the numerical columns list (containing dst_port) and a OneHotEncoder with sparse_output set to False and handle_unknown set to "ignore" applied to the categorical columns list (containing the remaining nine features). The ColumnTransformer is configured with remainder set to "drop" to ensure that any unexpected columns in the input DataFrame are silently excluded.
The fourth step is model instantiation and pipeline assembly. An IsolationForest estimator is instantiated with n_estimators set to 200, contamination set to 0.05, and random_state set to 42. The preprocessing ColumnTransformer and the IsolationForest estimator are combined into a scikit-learn Pipeline with two named steps: "preprocessor" and "model". The Pipeline object encapsulates the complete transformation and estimation sequence.
The fifth step is model fitting. The Pipeline's fit method is called on the feature DataFrame. Internally, this invokes the preprocessor's fit_transform method on the input data (which fits the StandardScaler to the numerical columns and the OneHotEncoder to the categorical columns, then transforms the data) and then passes the transformed output to the IsolationForest's fit method. The IsolationForest builds an ensemble of 200 isolation trees, each constructed by randomly selecting a feature and a random split value within the observed range of that feature, recursively partitioning the transformed feature space. The depth at which each training record is isolated across all trees is recorded and used to compute the normalised anomaly score for that record.
The sixth step is model serialisation. The fitted Pipeline object is serialised to ml/models/isolation_forest.pkl using joblib.dump. Joblib is used in preference to Python's standard pickle module because joblib handles large NumPy arrays (such as those produced by one-hot encoding of high-cardinality categorical features) more efficiently in terms of both serialisation speed and file size.
The contamination parameter of 0.05 represents the prior probability that any given record in the training dataset is anomalous. Setting this parameter determines the threshold of the anomaly score below which a record is classified as normal by the decision_function method. A value of 0.05 was selected based on the expected proportion of attack traffic in the lab dataset, given that the training data includes traffic from both baseline (non-attack) periods and the initial phases of attack scenarios. If the system is retrained on a dataset known to contain exclusively baseline traffic, a lower contamination value (for example, 0.01) would be more appropriate.
6.7 Inference Pipeline
The inference pipeline is implemented in ml/infer.py and is responsible for applying the trained model to new log records to produce anomaly predictions.
The pipeline begins by loading the serialised model from ml/models/isolation_forest.pkl using joblib.load. This restores the complete Pipeline object, including the fitted ColumnTransformer and the fitted IsolationForest, in a state identical to the end of training. No reprocessing of the training data is required.
New log records are read from the parsed_logs.csv file (or from a separately specified input file for streaming inference). The same ten feature columns used during training are selected from the input DataFrame. The Pipeline's predict method is called on the input feature DataFrame, which internally applies the fitted ColumnTransformer's transform method (using the already-fitted StandardScaler and OneHotEncoder parameters, not refitting them) and then calls the IsolationForest's predict method on the transformed data. The predict method returns a NumPy array of binary labels: -1 for anomalous records and 1 for normal records.
In addition to the binary prediction, the Pipeline's decision_function method is called on the same input data, returning a continuous anomaly score for each record. The decision_function output is the negated average path length across all isolation trees, normalised such that values closer to -0.5 indicate strong anomalies and values closer to 0.5 indicate normal observations. These continuous scores are preserved alongside the binary predictions in the output to allow threshold tuning and human review without rerunning the model.
Records for which the binary prediction is -1 (anomalous) are filtered from the output DataFrame and written to ml/data/anomalies.csv with their source IP, anomaly score, and timestamp. For each anomalous record, the inference module sends an HTTP POST request to the FastAPI control service at http://localhost:8000/ban with a JSON payload containing the src_ip field and the anomaly score. The POST request is sent using the Python requests library with a timeout of five seconds, and any connection error or timeout is logged to the inference module's error log without halting the inference process.
6.8 FastAPI Control Service
The control service is implemented in control-api/app.py using the FastAPI web framework. FastAPI was selected over Flask and Django because it provides automatic input validation using Pydantic models, asynchronous request handling, and automatic OpenAPI documentation generation, all of which are useful for an API that must handle concurrent ban requests from the inference module while remaining responsive to status queries from the operator.
The service defines a Pydantic model called BanRequest with two fields: ip_address (a string validated to match an IPv4 address pattern) and anomaly_score (a float). This model is used as the request body type for the /ban endpoint, ensuring that malformed requests are rejected before any API call to OPNsense is made.
The /ban endpoint is defined as an async POST handler. When a valid BanRequest is received, the handler first checks whether the ip_address is already present in the in-memory ban set to avoid duplicate API calls for the same source. If the address is new, the handler constructs an HTTPS POST request to the OPNsense alias management endpoint at https://[opnsense-ip]/api/firewall/alias_util/add/AUTO_BAN_IPS with a JSON body containing the address value. The OPNsense API requires HTTP Basic Authentication using an API key and API secret pair; these credentials are read from environment variables populated from the .env file using the python-dotenv library at application startup. After the add request, the handler sends a second HTTPS POST to the alias reconfiguration endpoint at https://[opnsense-ip]/api/firewall/alias_util/update to apply the change to the live firewall rule set without requiring a full configuration reload.
After the OPNsense API calls are complete, the handler appends a record to the local audit log file (control-api/audit_log.csv) containing the ip_address, anomaly_score, the HTTP status codes returned by both OPNsense API calls, and the UTC timestamp of the ban decision. This audit log is the operator's authoritative record of all automated enforcement actions taken by the system.
The /status endpoint is defined as an async GET handler. It queries the OPNsense alias utility endpoint at https://[opnsense-ip]/api/firewall/alias_util/list/AUTO_BAN_IPS and returns the current list of IP addresses in the alias as a JSON array. This endpoint is used by the Grafana dashboard to display the current effective block list.
6.9 Automated Enforcement Mechanism
The automated enforcement mechanism is the end-to-end pipeline that connects anomaly detection to firewall enforcement without human intervention. Understanding its operation requires tracing the flow of information across the system components.
When a network event occurs in the lab environment, it produces one or more log entries: a filterlog line in OPNsense, and possibly a Suricata alert in eve.json if a signature is matched. The parse_logs.py module reads these entries and appends normalised records to the parsed DataFrame. The infer.py module processes these records through the trained Isolation Forest pipeline and scores them. If any record receives an anomaly prediction of -1, the infer.py module immediately sends a ban request to the FastAPI control service.
The control service receives the request, validates the input using the BanRequest Pydantic model, checks the in-memory ban set to avoid duplication, and issues two sequential HTTPS calls to the OPNsense REST API: one to add the source IP to the AUTO_BAN_IPS alias and one to trigger an alias reconfiguration. OPNsense processes the reconfiguration request by reloading the pf rule set with the updated alias table contents, a process that takes approximately one to two seconds. From this point forward, any packet originating from the banned IP address that reaches the OPNsense OPT1 interface is matched by the AUTO_BAN_IPS block rule and dropped at the packet filter level, before reaching any higher-layer processing.
The enforcement action is thus fully automated, traceable, and reversible. Reversibility is provided through the OPNsense web interface, where an administrator can remove an IP address from the AUTO_BAN_IPS alias and trigger a reconfiguration to restore connectivity. The audit log produced by the control service provides a complete record of every automated decision, including the anomaly score that triggered it and the API response that confirms it was applied.
6.10 Observability Dashboard
The observability layer consists of three components deployed using Docker Compose on the control node: Promtail, Loki, and Grafana.
6.10.1 Promtail Configuration
Promtail is configured using a YAML configuration file that defines two scrape jobs. The first job scrapes the OPNsense filterlog file at /var/log/opnsense/filterlog.log, applying a pipeline stage that extracts the action field from the comma-separated log line and adds it as a Loki label. This labelling allows Grafana queries to filter log lines by action (pass or block) without parsing every line in the query layer. The second job scrapes the Suricata eve.json file, applying a pipeline stage that parses the JSON and extracts the event_type and alert.severity fields as Loki labels. Promtail maintains a position file that records the last-read byte offset of each scraped file, ensuring that on restart it resumes from the correct position without duplicating or skipping records.
6.10.2 Loki Configuration
Loki is configured with a local filesystem storage backend, storing indexed log chunks in /var/loki/chunks on the control node. The retention period is set to 72 hours for the lab environment, which is sufficient to cover the duration of all experimental scenarios. Loki exposes its HTTP API on port 3100 and is accessed by both Promtail (for log ingestion) and Grafana (for log querying) using the default HTTP interface.
6.10.3 Grafana Dashboard
Grafana is configured with a Loki data source pointing to http://loki:3100 (using Docker Compose service discovery by name) and is accessible on port 3000 of the control node. The dashboard is provisioned using a JSON configuration file that defines four panels.
The first panel is a time-series graph titled "Firewall Event Volume" that plots the count of filterlog entries per minute over the selected time range, using the LogQL query count_over_time for the filterlog job. Spikes in this graph during attack scenarios are visually correlated with the start and end times of the attack scripts.
The second panel is a time-series graph titled "Suricata Alert Rate" that plots the count of eve.json records with event_type equal to "alert" per minute. This graph shows the signature-based detection activity alongside the volumetric traffic graph, allowing the operator to distinguish between high-volume traffic events that do and do not produce Suricata alerts.
The third panel is a table titled "Recent Anomaly Detections" that queries the Loki instance for log lines written by the inference module when an anomaly is detected. Each row in the table shows the source IP, anomaly score, and timestamp of the detection event. This panel updates in real time as the inference module processes new records.
The fourth panel is a table titled "Current Block List" that queries the /status endpoint of the FastAPI control service and displays the current contents of the AUTO_BAN_IPS alias. This panel confirms that automated enforcement decisions are reflected in the active firewall configuration and provides the operator with a current view of the blocked source set.
6.11 Containerised Lab Setup
The observability stack (Promtail, Loki, Grafana) and the FastAPI control service are deployed using Docker Compose on the Ubuntu control node. Docker Compose was selected for the deployment of these components because it provides declarative multi-container orchestration using a single YAML configuration file, ensuring that the deployment is reproducible and that the inter-service dependencies (Grafana depending on Loki, Promtail depending on Loki) are managed automatically.
The docker-compose.yml file defines four services: loki, promtail, grafana, and control-api. The loki service uses the official grafana/loki image and mounts a local directory for chunk storage. The promtail service uses the official grafana/promtail image and mounts both the Promtail configuration YAML and the log directories (/var/log/opnsense and /var/log/suricata) as read-only bind mounts so that Promtail can access the log files without requiring the log-writing processes to be containerised. The grafana service uses the official grafana/grafana image and mounts a provisioning directory containing the Loki data source configuration and the dashboard JSON. The control-api service builds from the Dockerfile in the control-api directory, which uses a Python 3.11 slim base image, installs the dependencies listed in requirements.txt (fastapi, uvicorn, httpx, python-dotenv, pydantic), and runs the FastAPI application using uvicorn.
All four services are connected to a shared Docker bridge network named brb-network, which provides DNS-based service discovery between containers. The control-api service mounts the .env file from the host filesystem as a read-only bind mount, making the OPNsense API credentials available to the application without embedding them in the Docker image. Port mappings are configured to expose Grafana on host port 3000, Loki on host port 3100, and the control API on host port 8000, making these services accessible from other machines on the LAN segment.
The OPNsense appliance and the Kali attacker machine are not containerised. OPNsense runs as a virtual machine on the hypervisor (VirtualBox or similar), and the Kali machine runs as a separate virtual machine. The Docker Compose setup runs on the Ubuntu control node virtual machine, which is connected to the LAN segment (192.168.50.20). This separation between the containerised sidecar components and the firewall/attacker VMs reflects the sidecar architecture principle: the intelligence layer is external to the enforcement layer and communicates with it only through its documented API.
6.12 Attack Simulation Methodology
To validate the detection and enforcement capabilities of the BRB-NGFW system, ten attack scenarios are implemented as shell scripts in the kali-scenarios directory, each executed from the Kali attacker machine at 192.168.60.10 targeting the Ubuntu victim at 192.168.50.10.
The TCP SYN burst scenario (tcp_syn_burst.sh) uses hping3 to generate a high-rate stream of TCP SYN packets directed at port 80, simulating a SYN flood denial-of-service attack. The web scanning scenario (web_scan.sh) uses nmap with an aggressive scan profile to enumerate open ports and service versions on the target. The SSH brute force scenario (ssh_bruteforce_sim.sh) uses Hydra with a dictionary of common credentials to attempt repeated SSH authentication on port 22. The SQL injection scenario (sql_injection_sim.sh) uses sqlmap against a web application endpoint to enumerate database structures through HTTP parameter injection. The UDP flood scenario (udp_flood.sh) uses hping3 in UDP mode to generate high-volume UDP traffic directed at multiple ports. The ICMP flood scenario (icmp_flood.sh) uses hping3 to generate a high-rate ICMP echo request stream. The FIN scan scenario (fin_scan.sh) uses nmap with the -sF flag to perform a TCP FIN scan, which probes for open ports by exploiting RFC 793 behaviour. The slow HTTP scenario (slow_http.sh) uses slowhttptest to open a large number of HTTP connections and keep them open by sending headers slowly, exhausting the web server's connection pool. The credential stuffing scenario (credential_stuffing_http.sh) submits a large number of HTTP POST login requests using a list of username-password combinations sourced from publicly available credential datasets. The command injection probing scenario (cmd_injection_probe.sh) sends HTTP requests with shell metacharacters embedded in query parameters to probe for command injection vulnerabilities in the web application.
Each scenario is executed for a defined duration (typically 60 to 120 seconds) and the system's detection and enforcement responses are recorded by the Grafana dashboard and the audit log. The ground truth for each scenario is established by the known start and end time of each script execution, allowing the detection latency and false negative rate to be computed precisely.
